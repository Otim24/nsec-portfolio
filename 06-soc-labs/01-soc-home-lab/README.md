## 🧭 Executive Summary

This project simulates a real-world **Security Operations Center (SOC)** focused on detection engineering, log analysis, and threat investigation.

The lab integrates **Splunk (SIEM)** and **Wazuh (EDR)** to detect and analyze adversary activity generated through controlled attack simulations.

### Key Outcomes

- Built a segmented SOC lab with Active Directory
- Engineered a working log pipeline with proper parsing
- Simulated attacker activity (Nmap + SMB brute force)
- Developed detection logic using Splunk SPL
- Performed alert triage and false-positive analysis
- Correlated logs across SIEM and EDR platforms

### 🧠 Splunk Threat Hunting — Failed Logons (EventCode 4625)
![Splunk Search](screenshots/splunk-search.png)

**Analysis:**
This query surfaces failed authentication attempts across the environment. A high volume of 4625 events initially suggests potential brute-force activity.

**SOC Insight:**
EventCode 4625 is a primary indicator for:
- Password spraying
- Brute force attacks
- Misconfigured services

Further investigation is required to determine legitimacy.

---

### 🎯 Targeted Account Analysis
![Splunk Search](screenshots/splunk-search.png)

**Analysis:**
Aggregation of failed logins reveals that the **Administrator account** is the most targeted.

**SOC Insight:**
High-frequency failures against privileged accounts are a strong signal of:
- Credential attacks
- Automated attack tooling

This would normally trigger alert escalation in a production SOC.

---

### 🧪 False Positive Triage (Critical SOC Skill)
![Splunk Search](screenshots/splunk-search.png)

**Analysis:**
Deep inspection of a 4625 event shows:
- Logon Type: 7 (local unlock)
- Source IP: 127.0.0.1 (local system)
- Process: svchost.exe

**Conclusion:**
This activity is **not malicious**, but generated by local system behavior.

**SOC Insight:**
This demonstrates a core analyst skill:
> Not all alerts are attacks — context determines threat validity

Failure to perform this step leads to alert fatigue and wasted response effort.

## ⚙️ Phase 1 — Infrastructure & Log Pipeline

### Key Engineering Outcome

A fully functional log pipeline was established:

Windows → Splunk Forwarder → pfSense → Splunk Index → Analyst Queries

### Critical Configuration

```ini
[WinEventLog://Security]
index = main
sourcetype = WinEventLog:Security

[WinEventLog://System]
index = main
sourcetype = WinEventLog:System
```

**Impact:**
Ensures structured logs with searchable fields required for detection engineering.

## 🧠 Phase 3 — Detection Engineering

### Core Detection Query

```splunk
index=main sourcetype=WinEventLog:Security EventCode=4625
```

### Detection Logic (Brute Force)

```splunk
index=main sourcetype=WinEventLog:Security EventCode=4625
| bucket _time span=1m
| stats count by _time, Account_Name
| where count > 10
```
